# 技术选型草案

## 核心原则

`yiz-tunnel` 的核心转发部分以性能为主要目标。

第一阶段原则：

- 核心网络 IO、连接处理、转发流程尽量不使用外部依赖。
- 核心部分优先自行实现，便于控制性能、内存、状态和行为细节。
- 管理 API 性能要求不高，可以使用外部 HTTP 服务库。
- JSON 解析、管理 API、日志等非核心链路可按实际需要使用成熟依赖。

## 第一阶段依赖

已确认：

- 事件调度采用 `tokio`。
- JSON 解析使用 `serde_json`。
- 管理 API 使用 `axum`。
- TLS 后续使用 `rustls`。
- 请求头解析自行实现，可参考 nginx 源码。

## 模块边界

核心部分包括：

- HTTP 请求接收和解析。
- HTTP 请求头解析。
- HTTP route 匹配。
- 静态文件响应的核心处理流程。
- HTTP 代理转发。
- TCP 端口转发。
- 连接生命周期管理。
- upstream 状态和活动连接/请求计数。
- 热更新时的运行时对象切换。

非核心部分包括：

- 管理 HTTP API。
- JSON 配置读写。
- 日志落盘。
- 错误码和响应序列化。
- CLI 参数解析。

## 管理 API

管理 API 可以使用外部 HTTP 服务库。

选择标准：

- 稳定。
- 易维护。
- 与核心运行时边界清晰。
- 不强行影响核心 IO 模型。

具体库后续实现前再定。
