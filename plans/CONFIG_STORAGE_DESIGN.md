# 配置存储设计草案

## 定位

`yiz-tunnel` 的配置存储分为两类：

- 系统配置文件：用于描述程序自身如何启动和运行。
- 规则持久化文件：用于保存通过管理 API 创建和修改的业务规则。

系统配置文件与规则持久化文件不是同一类概念。

系统配置文件类似常见 Linux 程序配置，例如 nginx 通过启动参数指定配置文件位置。规则持久化文件则更接近程序运行过程中产生和维护的数据。

## 系统配置文件

系统配置文件名为：

```text
yiz-tunnel.json
```

启动程序时支持通过 `-c` 指定系统配置文件路径：

```text
yiz-tunnel -c /etc/yiz-tunnel.json
```

启动路径规则：

1. 如果传入 `-c`，使用 `-c` 指定的路径作为系统配置文件路径。
2. 如果未传入 `-c`，使用当前工作目录下的 `yiz-tunnel.json`。
3. 如果系统配置文件不存在，则按默认值生成系统配置文件。

默认系统配置文件只保存系统级配置，不保存 HTTP 服务规则和 TCP 转发规则。

系统配置文件至少应包含：

- 配置版本。
- 数据目录位置。
- 日志目录位置。
- 管理 HTTP 服务配置。
- 程序运行时基础配置。

示例：

```json
{
  "version": 1,
  "data-dir": "./data",
  "log-dir": "./logs",
  "admin": {
    "host": "127.0.0.1",
    "port": 9000
  },
  "runtime": {}
}
```

第一阶段确认：

- 系统配置文件先采用上述最小结构。
- 管理 HTTP 服务第一阶段不考虑鉴权。
- 默认数据目录为 `./data`。
- 默认日志目录为 `./logs`。
- 日志按类型分文件输出。

## 规则持久化文件

`http-server` 和 `tcp-forward` 的规则配置不直接写入系统配置文件。

它们应写入系统配置文件指定的数据目录中。

默认目录结构：

```text
data/
  http-server.json
  tcp-forward.json
```

如果 `data-dir` 下不存在 `http-server.json` 或 `tcp-forward.json`，启动时不自动生成规则文件。

规则文件只在用户通过管理 API 新增对应类型规则时生成。

如果按 Linux 常见部署方式运行，则可通过 `system-conf` 指定为：

```text
/var/lib/yiz-tunnel/
  http-server.json
  tcp-forward.json
```

两个规则文件分别保存：

- `http-server.json`：所有 HTTP 服务规则。
- `tcp-forward.json`：所有 TCP 端口转发规则。

第一阶段不按单个 `http-server` 或单个 `tcp-forward` 拆分文件。

## 文件职责

### yiz-tunnel.json

- 程序启动时读取。
- 保存系统级配置。
- 指定规则数据目录位置。
- 指定日志目录位置。
- 指定管理 API 配置。
- 第一阶段管理 API 不考虑鉴权。
- 不保存业务规则。
- 不保存运行状态。

### http-server.json

- 由管理 API 创建、修改、删除。
- 保存所有 `http-server` 规则。
- 只保存用户期望配置。
- 不保存运行时 `status`。
- 不保存活动连接数、请求数、错误统计等运行数据。

### tcp-forward.json

- 由管理 API 创建、修改、删除。
- 保存所有 `tcp-forward` 规则。
- 只保存用户期望配置。
- 不保存运行时 `status`。
- 不保存活动连接数、流量统计、错误统计等运行数据。

## 配置与运行状态分离

配置文件只保存规则和期望状态，例如：

- `enabled`
- 监听地址和端口。
- 路由规则。
- upstream 配置。
- TCP 转发目标。
- 超时、限制等配置。

配置文件不保存运行状态，例如：

- `status`
- 活动请求数。
- 活动连接数。
- 端口绑定失败原因。
- 最近错误。
- 启动时间。
- 流量统计。

运行状态由程序内存维护，并通过管理 API 对外暴露。

## 管理 API 写入模型

管理 API 修改配置时采用：

```text
先落盘，再调整运行时
```

流程：

1. 管理 API 收到用户期望配置。
2. 校验配置结构。
3. 生成软件内部 ID。
4. 事务化写入对应规则持久化文件。
5. 写入成功后，对单个对象执行局部 reload。
6. 如果运行时应用失败，配置仍然保留，运行状态记录失败原因。

示例：

用户新增一个监听 `7000` 端口的 `http-server`，但 `7000` 端口已被其它程序占用。

处理方式：

1. 如果配置结构合法，写入 `http-server.json`。
2. 局部 reload 时绑定端口失败。
3. 该 `http-server` 的运行状态进入失败状态。
4. 管理 API 返回失败原因。
5. 用户关闭占用 `7000` 端口的程序后，可重新启动该 `http-server`，无需重新提交完整配置。

## 校验策略

配置校验分为两类：

### 结构错误

结构错误直接拒绝，不落盘，不调整运行时。

例如：

- JSON 结构不合法。
- 必填字段缺失。
- 字段类型错误。
- 端口值非法。
- ID 冲突。
- route 指向不存在的 upstream group。
- action 类型不支持。
- upstream host 格式非法。

### 运行环境错误

运行环境错误允许落盘，但运行时进入失败状态。

例如：

- 端口被占用。
- 绑定地址不可用。
- 静态文件目录不存在或不可访问。
- upstream 暂时不可达。
- 日志目录暂时不可写。

## ID 生成

ID 由软件生成，不由用户传入。

ID 应全局唯一。第一阶段可采用 UUID，并可增加对象类型前缀以便排查。

示例：

```text
hs_xxx  http-server
rt_xxx  route
up_xxx  upstream
tf_xxx  tcp-forward
```

## 事务写入

规则持久化文件写入应采用事务化方式，避免文件损坏。

写入时可使用：

```text
http-server.json
http-server.json.tmp
http-server.json.bak
```

基本流程：

1. 读取当前规则文件。
2. 应用单个对象变更。
3. 校验新结构。
4. 写入临时文件。
5. 确认临时文件可重新解析。
6. 备份当前正式文件。
7. 用临时文件替换正式文件。
8. 对单个对象执行局部 reload。

## reload 策略

管理 API 只允许单个对象操作，不允许批量操作。

因此 reload 也按单个对象执行：

- 创建单个 `http-server` 时，只加载该 `http-server`。
- 更新单个 `http-server` 时，只替换该 `http-server`。
- 删除单个 `http-server` 时，只停止该 `http-server`。
- 创建、更新、删除 `tcp-forward` 时，只影响对应 `tcp-forward`。

规则文件按模块整体存储，运行时按对象局部 reload。

## 手工修改文件

第一阶段不考虑运行中手工修改配置文件。

程序只在启动时读取系统配置文件和规则持久化文件。

如果规则持久化文件不存在，启动时按空规则处理，但不主动创建文件。

运行中的配置变更只通过管理 API 产生。

第一阶段不实现：

- 文件变更监听。
- 手工修改自动 reload。
- 外部编辑配置文件后的冲突处理。
